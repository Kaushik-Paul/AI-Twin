# AGENTS.md

Guidance for coding agents working in this repository.

## Project Shape

- This is an AI digital twin for Kaushik Paul.
- `frontend/` is a Next.js 16 / React 19 static export app. The main chat UI is in `frontend/components/twin.tsx`.
- `backend/` is a Python 3.12 FastAPI app. The API app is `backend/main/server.py`; Lambda wraps it through `backend/lambda_handler.py`.
- `backend/main/chat_agents.py`, `backend/main/context.py`, and `backend/main/evaluation.py` contain the agent, prompt assembly, tools, and response evaluator.
- `backend/data/` contains source material for the twin: resume, LinkedIn export, summary, style notes, and structured facts. Treat this as user-owned profile data.
- `memory/` is for local conversation memory. Keep only `memory/.gitkeep` tracked.
- `terraform/` defines AWS Lambda, API Gateway, S3, CloudFront, and related deployment resources.
- `scripts/deploy.sh` and `scripts/destroy.sh` perform real AWS operations. Do not run them unless the user explicitly asks.

## Local Development

From the repository root:

```bash
# Backend
cd backend
uv run uvicorn main.server:app --reload --port 8000
```

```bash
# Frontend
cd frontend
npm install
npm run dev
```

For local frontend development, use `frontend/.env.local` with:

```bash
NEXT_PUBLIC_API_URL=http://localhost:8000
NEXT_PUBLIC_BASE_STAGE_NAME=
NEXT_PUBLIC_CHAT_ENDPOINT_API_KEY=local-dev
```

For local backend development, set `OPENROUTER_API_KEY` and `CORS_ORIGINS=http://localhost:3000` in the shell or a root `.env`.

Use `terraform validate` only after `terraform init` has already been run for the intended backend/workspace.

## Implementation Notes

- Keep backend imports compatible with both local FastAPI execution from `backend/` and Lambda packaging. `lambda_handler.py` imports `backend.main.server`, while local development runs `main.server`.
- If you add backend dependencies, update both `backend/pyproject.toml` / `backend/uv.lock` and `backend/requirements.txt`, because Docker-based Lambda packaging installs from `requirements.txt`.
- Do not commit generated backend artifacts such as `backend/lambda-package/`, `backend/lambda-deployment.zip`, `__pycache__/`, or `.ruff_cache/`.
- The frontend is configured for static export in `frontend/next.config.ts`; avoid adding features that require a Next.js server runtime unless you also change the deployment model.
- Frontend environment variables beginning with `NEXT_PUBLIC_` are exposed to the browser. Never put private secrets there.
- API key enforcement differs locally and in production. Locally, `CHECK_CHAT_API_KEY` defaults to false; production may validate `x-api-key`.
- Preserve the current UI pattern: chat-first, compact, theme-token driven CSS variables in `frontend/app/globals.css`, and lucide icons for interface controls.

## Agent And Prompt Safety

- Do not invent facts about Kaushik. Update `backend/data/facts.json`, `summary.txt`, `style.txt`, or the source PDFs only when the user explicitly provides new profile information.
- Keep tool behavior aligned with `backend/main/chat_agents.py`: contact capture uses Mailjet, resume delivery sends the resume PDF, and unknown questions are logged for follow-up.
- If you change the prompt in `backend/main/context.py`, update evaluator expectations in the same area when needed so guardrails stay consistent.
- Avoid logging secrets, API keys, email credentials, or private user conversation content beyond the existing memory behavior.

## Infrastructure And Deployment

- Treat `scripts/deploy.sh`, `scripts/destroy.sh`, `backend/deploy.py`, `terraform apply`, and `terraform destroy` as production-impacting commands. Ask for explicit confirmation before running them.
- `backend/deploy.py` uses Docker and AWS APIs and currently targets Lambda deployment behavior directly; inspect it before changing or invoking deployment flows.
- Keep Terraform variables and outputs consistent with the frontend env values generated by `scripts/deploy.sh`.
- Do not commit local `.env` files, Terraform state, generated tfvars, or AWS credentials.

## General Coding Style

- Match the style of the file you are editing. Python uses 4-space indentation; TypeScript/React style varies by file, so keep nearby formatting consistent.
- Prefer focused changes over broad refactors.
- Use structured JSON handling for conversation and profile data instead of ad hoc string parsing.
- Keep comments sparse and useful, especially around deployment behavior, prompt logic, and cross-environment path assumptions.
